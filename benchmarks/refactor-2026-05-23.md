## What changed, in order

### 1. Lazy stdlib `base.pkl` resolution

The CLI was unconditionally reading
`third_party/apple-pkl/stdlib/base.pkl` (~140 KB) at startup so the
reflect intrinsics had it available, even on fixtures that never
touch `reflect.*` (every bench fixture, in fact).

Fix: `configure_stdlib_base_paths(loader)` registers a closure; the
getter reads + caches on first hit. The negative case (no candidate
path) is also cached, so subsequent calls are O(1). This shaved
~0.7 ms off the constant startup floor across every fixture.

### 2. Memoize class-default purity verdict

`class_default_is_memoizable` re-walked the property / method bodies
of memoizable classes on every call site. The negative case was
already cached via `unsafe_classes`; mirror that with a
`safe_classes` map keyed by type name so the recursive purity walk
runs once per class per `eval_source`, not once per reference.

### 3. Strip captured `source` in place

`eval_class_default_members_seen`'s trailing strip pass allocated a
second `Array[ValueMember]` and rebuilt every struct. Most slots
come from `merge_value_members` with `source` already `None`;
update in place and only rewrite the override-derived slots.

Also folded a duplicate `lookup_class_binding` call (property /
method capacity probe).

### 4. Refactor: extract Protobuf renderer

`eval_expr.mbt` grew to 13150 lines, well past the 2k-per-file
target the refactor skill suggests. The Protobuf-renderer block (24
functions, ~720 lines) was self-contained: only the entry point and
error-detection helper were called from the wider renderer dispatch,
and the rest cross-referenced internally for the type-aware
union-discriminant emission Apple Pkl's `protobuf.Renderer` does.

Moved to `eval_render_protobuf.mbt`. Same package so no visibility
changes. `collect_class_properties_in_order` stayed in eval_expr.mbt
because `eval_stdlib_scalar.mbt` also calls it.

eval_expr.mbt: 13150 → 12427 lines.

### 5. Reflect dead code + reserved-keyword cleanup

`eval_stdlib_reflect.mbt` had:
- `StdlibTypeAliasMeta` struct (4 unread fields)
- `aliases : Map[...]` on `StdlibReflectDb` (only written, never read)
- `is_abstract` field on `StdlibClassMeta` (unused — value pulled
  directly off `ClassDecl`)
- `declarations` field on `StdlibReflectDb` (orphaned scaffold)
- `stdlib_decl_header_line` (typealias-only helper, no callers)

All from the initial reflect bring-up. Removed; saved minor build
cost on every `reflect.Class()` call site that triggers the lazy db
load.

Also renamed `method` → `method_value` in
`gold_replace_method_display_uri` to clear the reserved-keyword
warning.

`moon check` now warning-free.

## What's still on the table

`pkspec-test-schema` at 2.11× pkl is the remaining real gap. The
top sample buckets per the prior retrospective are GC / allocator
(`moonbit_drop_object`, `malloc`, `_platform_memmove`) and
structural `Value` equality (`moonbit_val_array_equal`). Further
gains there would need either:

- Per-call-site Array pooling on the class-default synthesis path
  (defaults / base_values / own_methods scratch arrays).
- Cheaper structural-equality fast paths (length / first-element
  pre-checks before full `Value` recursion).
- A separate compact representation for ClassDefault `ValueMember`
  arrays so the strip / merge pipeline operates on a smaller
  per-slot footprint.

Lower bang-per-buck than the previous round's leaf-string work, and
the noise floor on the hyperfine bench (~0.5 ms / fixture run-to-run)
makes single-percentage gains hard to attribute confidently.
