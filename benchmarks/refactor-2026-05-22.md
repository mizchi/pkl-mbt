# Performance refactor — 2026-05-22

Iterative optimization on the eval / reflect hot path, validated
against Apple Pkl 0.31.1 native binary on a 1643-line typed schema
(`pkspec/pkl/Test.pkl`) and the upstream `apple-pkl/stdlib/base.pkl`.

## Tooling

- `scripts/bench-vs-pkl.sh` — hyperfine-based end-to-end comparison
  between `mpkl eval` (native AOT) and `pkl eval` (native AOT). Same
  fixtures, same shell-less invocation, fairness-guarded by a
  pre-bench `diff` pass that refuses to time fixtures whose outputs
  diverge.
- `moon bench --target js` — per-stage breakdown (tokenize / parse /
  typecheck / eval) for the heavy fixtures, used to localise
  regressions and pinpoint hot leaves.
- `sample` (macOS) on the mpkl release binary — exact stack traces
  from inside a hot eval, used to find the actual leaf doing work
  after each round.

## What changed, in order

### 1. Stack overflow on `apple-pkl/stdlib/base.pkl`

`synthesize_default_for_type` → `apply_class_defaults_for_type` →
`eval_class_default_members` recursed without threading the cycle
guard. Mutually-recursive class types (`class A { b: B }`,
`class B { a: A }`) blew the call stack instead of returning the
empty-default shape Apple Pkl produces.

Fix: added `_seen` variants that thread the in-progress class chain
through the recursion; the existing `eval_class_default_members_seen`
already had the guard, but the synthesise / apply pair was resetting
it.

### 2. Reflect source scans were O(L × N) per module

`reflect_module_metadata_value` walks every class and every member;
for each (class, member, kind) it called
`reflect_class_member_decl_line_from_source` and
`doc_comment_before_class_member`, each of which split the module
source into lines and walked them looking for one match. With
`pkspec/Test.pkl` (1643 lines, 49 classes, ~500 reflect queries),
the leaves spent ~70% of total time in `line_declares_reflect_name`
→ `boyer_moore_horspool_find` (confirmed by `sample`).

Fix: introduced `ReflectSourceIndex` — a per-source cache that
pre-splits lines and builds `(class, member, kind) → line index` and
`(name, kind) → line index` maps in a single pass. Cached at top
level keyed by source identity (LRU cap = 4); doc-comment scanners
walk backwards from the cached line index instead of re-splitting.

### 3. Typealias bindings rebuilt every call

`eval_type_alias_bindings(declarations)` re-seeded the 9 builtin
`pkl:base` aliases and walked every user typealias on every call —
182 / 5000 samples on apple-pkl/stdlib/base.pkl. Within a single
`eval_source` pass the declarations array is the same reference
repeatedly.

Fix: a 4-entry `physical_equal` cache keyed by the declarations
array. `eval/apple-pkl-stdlib-base` 9.80 s → ~6 s.

### 4. Linear-walk shadowing lookups

`lookup_class_binding` / `lookup_eval_type_alias` / `lookup_value` /
`lookup_member` / `lookup_value_member` / `find_binding` all used
the same "forward-walk, keep overwriting `found`" pattern. The
intent — "last shadowing binding wins" — was correct, but it always
paid O(N) even when the latest binding (almost always the answer
on hot paths) sat at the tail.

Fix: reverse-walk + early-break preserves the same semantics at
typical-O(1).

### 5. Type-name parsers running boyer-moore on hot path

`pkl_constrained_type_paren_index` walked every char of every type
name looking for the first `(`; `split_top_level_union_choices`
allocated a StringBuilder per call even for plain names like `Int`.
The reflect / synthesize-default paths hit these thousands of times
per module.

Fix: fast paths that bail before the per-char loop when no `(` / `|`
is present, plus `buf.write_char(c)` instead of
`buf.write_string(c.to_string())` which allocated a transient one-
char String per iteration.

### 6. Collection-class detection ran 9 prefix checks

`synthesize_default_for_type_seen` checked `name == "Listing"`,
`name.has_prefix("Listing<")`, then the same for Mapping / Set / Map
/ List / Collection — up to 9 has_prefix scans per call. Every
builtin starts with L / M / S / C, so a first-char match cuts the
common (non-collection) case to a single char compare.

Fix: `match name[0]` dispatch. `eval/apple-pkl-stdlib-base` JS
bench 5.96 s → 4.78 s (~20% faster).

### 7. Class-env / typealias-env still linearly scanned

After reverse-walk landed, `lookup_class_binding` was still 95 /
5000 samples and `lookup_eval_type_alias` 65. Both arrays are
append-only during a single eval; turning each into a Map keyed by
name (with a physical-identity + length staleness check) drops both
out of the top-25 sample bucket.

### 8. Single-char `String.find` / `has_prefix` / `has_suffix`

MoonBit's pattern matchers route through
`brute_force_find` / `boyer_moore_horspool_find` even for one-char
patterns. The reflect / type-name paths used them heavily
(`name.find("(")`, `text.find(",")`, `name.has_prefix("(")`,
`name.has_suffix("?")`, etc).

Fix: `string_index_of_char` / `string_contains_char` /
`string_starts_with_char` / `string_ends_with_char` byte-loop
helpers and route the hot-path callers through them.
`brute_force_find` drops from 131 → ~26 samples;
`boyer_moore_horspool_find` from 92 → ~61.

## What's left

Top remaining samples after this round:

| samples | function | notes |
| ---: | --- | --- |
| 287 | `moonbit_drop_object` | GC pressure from Array.push and short-lived strings |
| 115 | `malloc` | same |
| 99 | `moonbit_val_array_equal` | structural Value equality (deferred-error / Listing-equality paths) |
| 60–62 | `eval_class_default_members_seen` | the function itself, called many times — would need memoization |
| 60 | `Hash.hash[String]` | from the Map caches we added |
| 50 | `_platform_memmove` | mostly Array.push growth |

The GC / allocation pressure is the next big target — would need
either Array pre-sizing on hot construction paths or pooled buffers
for transient builders. Lower bang-per-buck than the leaf-string
work this pass focused on.

`basic-as` / `basic-is` / `basic-rawstring` / `basic-new` still sit
slightly above 1.0× pkl — same family of typed-collection
retention, regex literal handling, and multi-line raw-string
processing. Not investigated yet; would benefit from a fresh sample
specifically against those fixtures rather than stdlib/base.pkl.

`eval/apple-pkl-stdlib-base` still takes ~4 s on JS bench. That's
evaluating the entire `pkl:base` stdlib as a user module, which
isn't a real workload — the stdlib is normally consumed via
`import`, not evaluated. Lower priority than user-facing fixtures
but useful as a stress fixture.
